# ADR-001: AryanMart Platform Architecture Decisions

**Status:** Accepted  
**Date:** June 2026  
**Decision Makers:** AryanMart Engineering / Product  
**Scope:** Phase 1 — Strategy, Architecture & Planning

---

## Context

AryanMart Pvt. Ltd. requires an enterprise-grade B2B+B2C procurement platform specializing in telecom, networking, electrical, CCTV, IT hardware, and government/defense supply chains. The platform must support standard e-commerce, wholesale pricing, RFQ workflows, tender-oriented sales, rate contracts, and institutional procurement — not a generic marketplace template.

This ADR records the foundational architecture decisions that govern all subsequent implementation phases.

---

## Decision 1: Platform Identity — Procurement-Native Specialist

**Decision:** Build a category-specialist procurement portal, not a general-purpose marketplace.

**Rationale:**
- Primary differentiator is telecom + network + electrical + defense supply chain expertise
- Target users are procurement officers, system integrators, government buyers, and technical professionals
- UX promise: find, quote, and order in under 3 minutes
- Platform tone: trustworthy, institutional, precise — not consumer-gimmicky

**Consequences:**
- IA, search facets, and PDP specs prioritize technical attributes (HSN, voltage rating, certifications)
- RFQ and tender flows are first-class, not add-ons
- Design language ("Institutional Precision") reinforces government readiness

---

## Decision 2: Frontend — Next.js 14 App Router

**Decision:** Use Next.js 14 with App Router as the primary frontend framework.

**Rationale:**
- SSR/ISR for SEO-critical catalog and category pages
- React Server Components for performance on large product listings
- Strong ecosystem for B2B e-commerce patterns
- Vercel deployment aligns with edge-optimized delivery

**Alternatives Considered:**
- Remix — rejected due to smaller component ecosystem for enterprise UI
- SPA-only React — rejected due to SEO requirements for 10k+ SKU catalog

---

## Decision 3: Backend — Node.js REST + tRPC

**Decision:** Node.js with Express (REST) and tRPC for type-safe internal APIs.

**Rationale:**
- Familiar stack for rapid development
- tRPC provides end-to-end type safety between frontend and backend
- Scales horizontally on AWS EC2/ECS

---

## Decision 4: Data Layer — PostgreSQL + Redis + Elasticsearch

**Decision:**
- **PostgreSQL 15+** (via Prisma ORM) — primary transactional store
- **Redis 7+** — sessions, cart cache, rate limiting
- **Elasticsearch 8.x** — full-text and faceted product search

**Rationale:**
- PostgreSQL JSONB handles flexible product specs without schema churn
- ACID compliance required for orders, invoicing, and credit terms
- Elasticsearch supports faceted search across brands, certifications, voltage ratings, etc.

---

## Decision 5: Multi-Role Authentication

**Decision:** NextAuth.js + JWT + OTP (MSG91) with role-based access: `b2c | b2b | vendor | admin | govt`.

**Rationale:**
- GSTIN-linked B2B accounts require verified business identity
- Government users need isolated tender dashboard access
- Vendors require separate onboarding and fulfillment workflows

---

## Decision 6: Payment — India-First Gateways

**Decision:** Razorpay (primary) + PayU (fallback).

**Rationale:** UPI, NetBanking, Cards, EMI support for Indian market; B2B credit terms handled outside immediate payment flow.

---

## Decision 7: File Storage — AWS S3 + CloudFront

**Decision:** S3 for product images, catalogs, compliance documents; CloudFront CDN for global delivery.

---

## Decision 8: Hosting Split

**Decision:**
- Frontend: Vercel
- Backend API: AWS EC2/ECS
- CI/CD: GitHub Actions

**Rationale:** Edge-optimized Next.js on Vercel; scalable stateful backend on AWS.

---

## Decision 9: Competitive Positioning vs IndustryBuying

**Decision:** Borrow enterprise patterns (category-first nav, spec tables, B2B trust signals) but differentiate on telecom/defense specialization, government procurement tools, OFC depth, and GSTIN-first checkout.

**Explicit Non-Goals:**
- Do not clone IndustryBuying color scheme, layout proportions, or brand voice
- Do not build a generic industrial marketplace

---

## Decision 10: Implementation Phasing

**Decision:** Three-phase delivery:
1. **Phase 1** — Strategy, architecture, planning (this ADR and `/docs/`)
2. **Phase 2** — UI/UX system, design tokens, wireframes (no application code)
3. **Phase 3** — Full code implementation (blocked until Phase 1 & 2 reviewed)
